# Задача 9.2 — Изоляция вокала и подавление реверба

> **Источник:** [PLAN_DORABOTKI.md](PLAN_DORABOTKI.md), очередь 9.  
> **Цель:** выделить и усилить вокал; удалить фоновый шум и реверберацию; сделать высокие ноты плавными, без резкости и потрескивания.

---

## Варианты реализации

### A. Разделение источников (source separation)

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| **Demucs** (Meta) | Качество, поддержка vocal/other | PyTorch, большие модели, нужен GPU для скорости |
| **Spleeter** (Deezer) | Готовые 2/4/5 stems, простой API | Устаревает, качество ниже Demucs |
| **open-unmix** | Лёгкие модели, ONNX-экспорт | Отдельные модели под инструменты |
| **Без ML** | Нет зависимостей | Реалистично только простой mid/side или частотный срез |

Рекомендация для первого прототипа: **Demucs** (модель `htdemucs` или `htdemucs_ft`) с выходом только vocal; при отсутствии GPU — малая модель или вызов в отдельном воркере с таймаутом.

### B. Подавление реверба (dereverb)

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| **Спектральное вычитание** реверб-хвоста | Без ML, предсказуемо | Риск артефактов, нужна оценка IR/хвоста |
| **Простой задержка/фильтр** (early reflections) | Реализуемо в numpy/scipy | Слабо помогает при сильном ревербе |
| **Нейросетевой dereverb** | Лучшее качество | Отдельная модель, зависимости |

Рекомендация: сначала пробовать **A** (изоляция вокала); реверб на выделенном вокале часто субъективно меньше. При необходимости добавить лёгкий dereverb (спектральное сглаживание или гейт по энергии).

---

## Ограничения текущего стека

- **backend:** Python, numpy, scipy, soundfile, pyloudnorm, pydub. Нет PyTorch по умолчанию.
- **Деплой:** возможны ограничения по RAM/CPU; GPU не гарантирован.
- **Латентность:** мастеринг в реальном времени не требуется, но пакетная обработка не должна упираться в одну тяжёлую модель.

---

## Предлагаемые шаги

1. **Исследование:** поднять отдельный скрипт с Demucs (или Spleeter), прогнать на 2–3 тестовых файлах, оценить качество и время на CPU.  
   **Сделано:** скрипт `backend/run_isolate_vocal.py` и опциональные зависимости `backend/requirements-vocal-isolation.txt`. Запуск: `pip install -r requirements-vocal-isolation.txt`, затем `python run_isolate_vocal.py /path/to/track.wav [-o DIR]`. Модель по умолчанию — `htdemucs`, только стем vocals (`--two-stems vocals`).
2. **Опциональность:** вынести изоляцию вокала в отдельный модуль/эндпоинт (например, `POST /api/v2/isolate-vocal`), чтобы не нагружать основной пайплайн при отключённой фиче.  
   **Сделано:** эндпоинт `POST /api/v2/isolate-vocal` (файл → WAV вокала). При выключенной фиче возвращает 503. Сервис `app/services/vocal_isolation.py` вызывает Demucs через subprocess (без импорта torch в основном процессе).
3. **Конфиг:** флаг типа `ENABLE_VOCAL_ISOLATION=false` и путь к модели; при `false` не подгружать PyTorch/Demucs.  
   **Сделано:** `MAGIC_MASTER_ENABLE_VOCAL_ISOLATION=1`, `MAGIC_MASTER_DEMUCS_MODEL=htdemucs`. В `GET /api/health` в `features` добавлен ключ `vocal_isolation_enabled`.
4. **Интеграция в пайплайн:** если включено — после загрузки файла применять изоляцию вокала, затем передавать результат в существующую цепочку (румбл → Denoiser → De-esser → …).  
   **Сделано:** при `apply_vocal_isolation=true` в форме (и `ENABLE_VOCAL_ISOLATION=1`) в `_run_mastering_job_v2` сразу после загрузки аудио вызывается `isolate_vocal(data, filename)`; результат (WAV вокала) загружается и передаётся в румбл → Denoiser → De-esser → цепочка. Параметр добавлен в одиночный мастеринг и в batch. В UI: блок «Изоляция вокала» в «Доп. обработке» (виден при `vocal_isolation_enabled` в /api/health), чекбокс и передача `apply_vocal_isolation` в форме.
