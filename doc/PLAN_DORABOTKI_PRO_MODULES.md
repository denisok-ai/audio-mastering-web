## 1. Тишина после мастеринга

### 1.1 Симптомы

- Включены один или несколько инструментов: Spectral Denoiser, De-esser, Transient Designer, Parallel Compression, Dynamic EQ.
- После завершения мастеринга пользователь слышит тишину (или получает тихий/пустой файл).

### 1.2 Уже реализовано

- **validate_mastered_not_silent** в [backend/app/pipeline.py](backend/app/pipeline.py): при `max(|signal|) < 1e-7` выбрасывается `ValueError` с текстом «Обработка дала тишину. Отключите часть доп. настроек…».
- В [backend/app/routers/mastering.py](backend/app/routers/mastering.py) при любой ошибке в задаче выставляется `job["status"] = "error"`, `job["error"] = str(e)`.
- На фронте при `status === 'error'` выбрасывается исключение с `data.error`, в catch показываются тост и статус с полным сообщением; для текста с «тишину»/«Отключите» длительность тоста 7 с.

### 1.3 Возможные причины тишины

| Источник | Механизм | Действие |
|----------|----------|----------|
| **Spectral Denoiser** | При агрессивном strength + высокий noise_percentile оценка шума может «съесть» весь сигнал; min_gain=0.25 уже есть, но на граничных случаях возможна сильная редукция. | Ограничить снизу итоговый уровень (например по LUFS) или ослабить пресеты. |
| **Transient Designer** | Вызов при `transient_attack is not None` даже при 1.0/1.0; при значениях 0.1 (после clip) gain может сильно уменьшать сигнал. | Не вызывать Transient Designer, если оба параметра в зоне 1.0 (как в pipeline: \|attack−1\| < 0.02 и \|sustain−1\| < 0.02). |
| **Dynamic EQ** | В полосах gain_mult может давать сильный срез; при комбинации всех полос и тихом входе теоретически возможна сильная редукция. | Добавить нижнюю границу для gain_mult по полосе или байпас при очень низком уровне входа. |
| **De-esser** | Порог по умолчанию −10 dB и ratio 4:1: полоса 5–9 кГц срабатывает на большую часть вокала, а не только на сибилянты; сильное подавление «убивает» верха и присутствие. В UI допускается порог до −30 dB — при таком значении эффект срабатывает почти постоянно. | См. раздел 5. |
| **Порог валидации** | Сейчас порог 1e-7 по max(abs). Сигнал с max=1e-6 пройдёт проверку, но будет практически неслышимым. | Ужесточить проверку: например, дополнительно требовать integrated LUFS выше −70 dB или не экспортировать при max < 1e-5. |

### 1.4 Задачи по устранению тишины

| # | Задача | Описание |
|---|--------|----------|
| **1.4.1** | **Не вызывать Transient Designer при 1.0/1.0** | В [mastering.py](backend/app/routers/mastering.py) в `_run_mastering_job_v2`: вызывать `apply_transient_designer` только если `abs(transient_attack - 1.0) > 0.02` или `abs(transient_sustain - 1.0) > 0.02` (как в pipeline). Иначе не менять `mastered`. |
| **1.4.2** | **Ужесточить validate_mastered_not_silent** | В [pipeline.py](backend/app/pipeline.py): кроме условия `max(|signal|) < 1e-7`, при необходимости добавить проверку по integrated LUFS (например, не допускать экспорт при LUFS < −70 dB) или поднять порог до 1e-5, чтобы не отдавать «почти тихий» файл. |
| **1.4.3** | **Защита в Dynamic EQ** | В [apply_dynamic_eq](backend/app/pipeline.py): не применять срез полосы ниже разумного минимума (например gain_mult не ниже 0.2–0.3 в каждой полосе) или пропускать полосу при слишком низком уровне входа. |
| **1.4.4** | **Диагностика «виновника»** | Опционально: при включённом debug-режиме или по флагу логировать уровень сигнала после каждого PRO-модуля (или писать в job.message этап, на котором уровень упал ниже порога), чтобы быстрее выявлять, какой инструмент даёт тишину. |
| **1.4.5** | **Показ ошибки в пакетном режиме** | В [app.js](frontend/app.js) при пакетном мастеринге при `st.status === 'error'` показывать в тосте или в строке задания не только «Ошибка», но и `st.error` (полный текст от сервера), чтобы пользователь видел «Отключите часть доп. настроек…». |

## 4. De-esser — «убивает звук»

### 4.1 Симптомы

- De-esser включён (в т.ч. с порогом −10 dB по умолчанию).
- После мастеринга звук «убит»: глухой, без присутствия, верха подавлены, вокал теряет ясность.

### 4.2 Причина

- В [apply_deesser](backend/app/pipeline.py) порог задаётся в dBFS: при **−10 dB** (`thresh_lin ≈ 0.316`) срабатывание идёт на всё, что в полосе 5–9 кГц выше этого уровня. У многих записей вокала значительная часть энергии в этой полосе постоянно выше −10 dB, поэтому компрессия 4:1 применяется почти всё время, а не только на «сс», «фф», «тт».
- В UI слайдер порога: **min="-30" max="-5"** ([index.html](frontend/index.html)). При −30 dB порог очень низкий, де-эссер срабатывает на почти любой сигнал в полосе и сильно режет верха.
- Итог: полоса 5–9 кГц режется слишком сильно и слишком часто → субъективно «звук убит».

### 4.3 Задачи по исправлению De-esser

| # | Задача | Описание |
|---|--------|----------|
| **4.3.1** | **Нижняя граница gain_mult в apply_deesser** | В [pipeline.py](backend/app/pipeline.py) в `apply_deesser`: после вычисления `gain_mult` ограничить снизу, например `gain_mult = np.clip(gain_mult, 0.35, 1.0)` (или 0.4), чтобы полоса 5–9 кГц никогда не резалась сильнее чем примерно до 35–40% уровня. Это предотвращает «убивание» верхов при любых разумных пороге и ratio. |
| **4.3.2** | **Менее агрессивный порог по умолчанию** | Сменить дефолт порога с **−10 dB** на **−6 dB** (или −5 dB) в бэкенде и во фронте (атрибут `value="-10"` → `value="-6"` у слайдера, дефолт в [mastering.py](backend/app/routers/mastering.py) и в [app.js](frontend/app.js) при сборе формы). Тогда по умолчанию де-эссер будет срабатывать в основном на реальные пики сибилянтов, а не на весь вокал. |
| **4.3.3** | **Ограничить диапазон порога в UI** | Слайдер порога: вместо **min="-30"** задать **min="-18"** (или −15 dB), чтобы пользователь не выставлял порог, при котором эффект срабатывает на весь материал и «убивает» звук. При необходимости добавить подсказку: «Чем выше значение (ближе к 0), тем только на пики; при −18…−15 dB эффект мягче.» |
| **4.3.4** | **Снизить ratio по умолчанию (опционально)** | В [apply_deesser](backend/app/pipeline.py) рассмотреть снижение `ratio` по умолчанию с **4.0** до **2.5** или **3.0**, чтобы сжатие полосы было мягче и меньше риск пережатия при том же пороге. |
| **4.3.5** | **Подсказка в UI** | Рядом с ползунком «Порог» добавить краткий текст: «При слишком глухом звуке поднимите порог (например −6…−4 dB) или отключите De-esser.» |

### 4.4 Реализация (приоритет)

1. **Обязательно:** 4.3.1 (floor для gain_mult) — сразу убирает экстремальное «убивание» полосы.
2. **Рекомендуется:** 4.3.2 (дефолт −6 dB), 4.3.3 (min порога −18 dB).
3. **По желанию:** 4.3.4 (ratio), 4.3.5 (подсказка в UI).
