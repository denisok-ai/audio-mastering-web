# Аудит админ-панели Magic Master

> **Дата:** 2026-03  
> **Цель:** выявить нерабочие функции и составить план доработок по замечаниям пользователя.

---

## 1. Замечания пользователя

1. **Нет выхода из учётной записи** — кнопка «Выйти» не работает или не отображается.
2. **Не подключаются API ключи** — ввод и сохранение ключей (OpenAI, DeepSeek и т.д.) не дают эффекта.
3. **Настройки только визуал** — ощущение, что функции в настройках не используются и не работают.

---

## 2. Результаты проверки кода

### 2.1 Выход из учётной записи (logout)

**Текущая реализация (frontend/admin.html):**

- Кнопка «Выйти» создаётся в `checkAuth()` и вешается на `doLogout`.
- `doLogout()` очищает `_token`, удаляет `localStorage.admin_token`, вызывает `showLogin()` (показ overlay с формой входа).
- Overlay имеет `z-index: 400`, форма входа отображается поверх контента.

**Возможные причины проблемы:**

- Кнопка «Выйти» может быть не видна (узкий topbar, overflow, другой элемент перекрывает).
- После вызова `showLogin()` основной layout (сайдбар, контент) остаётся в DOM; при некорректной отрисовке или стилях overlay может не перекрывать контент.
- Нет явного скрытия основного интерфейса при выходе — только показ overlay.

**Рекомендация:** при выходе явно скрывать основной layout (например, `display: none` или `visibility: hidden`) и показывать только overlay входа; после входа — возвращать отображение layout. Убедиться, что кнопка «Выйти» всегда видна (не обрезается, не перекрывается).

---

### 2.2 API ключи (LLM: Подключение)

**Текущая реализация:**

- **Фронт:** форма «Подключение (LLM)» отправляет PATCH `/api/admin/settings` с телом `{ llm: { openai_api_key, deepseek_api_key, ... } }`. Ключи добавляются в payload только если поле не пустое.
- **Бэкенд:** `admin_settings_patch` принимает `SettingsUpdate`, в т.ч. `body.llm`; `_apply_settings_update` записывает ключи в `settings_store` (таблица `system_settings`).
- **Использование:** в `ai.py` функция `_get_llm_setting(key)` сначала читает значение из `settings_store.get_setting(key)`, при отсутствии — из config/env. То есть сохранённые в админке ключи должны подхватываться.

**Возможные причины проблемы:**

- Ошибка при PATCH (401/403/500) не разбирается пользователем — показывается общее «Ошибка» без деталей.
- После сохранения нет явной проверки (например, тестового запроса к LLM), поэтому непонятно, «подключились» ли ключи.
- В интерфейсе не отображается однозначный статус: «ключ сохранён и используется» / «ключ не задан» / «ошибка при сохранении».

**Рекомендация:** после успешного PATCH показывать сообщение «Ключи сохранены» и при необходимости обновлять блок статуса LLM (какой бэкенд активен, задан ли ключ). Опционально: кнопка «Проверить подключение» (лёгкий запрос к API). Улучшить отображение ошибок (текст ответа сервера).

### 2.3 Настройки (Общие, SMTP, YooKassa, Telegram и др.)

**Использование настроек в приложении:**

- **settings_store** используется в: `main.py` (maintenance_mode, temp_dir, feature_ai_enabled, feature_batch_enabled, feature_registration_enabled, alert_*), `ai.py` (API ключи, лимиты, промпты), `deps.py` (фичи), `routers/mastering.py` (default_target_lufs, max_upload_mb), `routers/misc.py`, `helpers.py` (allowed_extensions), `notifier.py` (SMTP, Telegram и др.).
- Значения читаются из БД (`system_settings`), при отсутствии — из config (env). То есть сохранённые в админке настройки реально используются без перезапуска (кроме тех, что в коде берутся только из config при старте).

**Возможные причины ощущения «только визуал»:**

- Часть параметров (например, CORS, JWT secret) по смыслу задаются через env и не переопределяются из админки или требуют перезапуска — это нигде не пояснено.
- После нажатия «Сохранить» нет явной обратной связи: какие именно параметры применились, что для вступления в силу нужно перезапустить сервер.
- Нет краткой справки в UI: «эти настройки применяются сразу», «эти — после перезапуска».

**Рекомендация:** в разделе «Настройки» добавить короткие подсказки: какие настройки действуют без перезапуска, какие — только после. После сохранения показывать список сохранённых секций и при необходимости предупреждение «для применения CORS/части параметров нужен перезапуск».

## 3. Дополнительные замечания

- **Единая точка входа:** админка использует `localStorage.admin_token`; основной сайт — `localStorage.mm_token`. Выход в админке не очищает `mm_token` (и наоборот). Если админ заходит в админку с того же браузера, где был под основным аккаунтом, это может путать. По желанию можно документировать или разделять сессии.
- **Валидация ключей:** при сохранении API ключей бэкенд не проверяет формат/доступность; полезно хотя бы проверять непустоту и при желании — делать тестовый запрос и показывать результат в админке.
